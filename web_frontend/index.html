PostgreSQL 地端關聯資料庫結構與預設內容 (schema.sql)

本系統在 PostgreSQL 地端資料庫中定義了三個核心實體表，用於儲存生理量測、設備原資料與病患摘要： 

-- 1. 病患基本資料表 (patients)
CREATE TABLE patients (
    id VARCHAR(50) PRIMARY KEY,       -- 'patient-01' (FHIR Patient.id)
    mrn VARCHAR(50) UNIQUE NOT NULL,  -- 'MRN998877' (病歷號)
    name VARCHAR(100) NOT NULL,       -- '蘇先生' (姓名)
    gender VARCHAR(10) NOT NULL,      -- 'male' (生理性別)
    birth_date DATE NOT NULL          -- '1991-12-25' (出生日期)
);

-- 2. 監測設備資料表 (devices)
CREATE TABLE devices (
    id VARCHAR(50) PRIMARY KEY,       -- 對應 FHIR Device.id
    serial_number VARCHAR(100) UNIQUE, -- 設備序號
    manufacturer VARCHAR(100),         -- 'FLIR', 'Infineon'
    model_number VARCHAR(100),         -- 'E53-Camera', 'MR60-Radar'
    device_type VARCHAR(100)          -- 'Thermal-Camera', 'Radar'
);

-- 3. 生理觀測指標資料表 (observations)
CREATE TABLE observations (
    id SERIAL PRIMARY KEY,
    patient_id VARCHAR(50) REFERENCES patients(id),
    device_id VARCHAR(50) REFERENCES devices(id),
    status VARCHAR(20) DEFAULT 'final',
    category VARCHAR(50),             -- 'vital-signs' 或 'survey'
    loinc_code VARCHAR(20),           -- LOINC Code (如 8867-4)
    recorded_at TIMESTAMP WITH TIME ZONE,
    value_numeric NUMERIC(6, 2),      -- 數值型資料 (如體溫 36.5)
    value_unit VARCHAR(20),           -- 單位 (Cel, /min)
    value_code VARCHAR(50),           -- 狀態型代碼 (SNOMED CT)
    value_display VARCHAR(100)        -- 狀態型顯示 (在床, 跌倒)
);

衛生福利部標準規格與 HAPI FHIR 資料庫儲存研析指引

1. 衛福部 TW Core IG 官方範例指引連結

以下為中華民國衛生福利部 (MHW) 與台灣 HL7 共同維護的官方 FHIR 範例與規範指引：

本專案開源 GitHub 倉庫 衛福部 TW Core IG 官方首頁 TW Core IG 官方範例索引手冊 病患 Patient Profile 結構定義與規範 生命徵象 Observation Profile 規格

本系統導出的 JSON 與 API 對接格式均精確符合上述官方實作指引之規範，能無縫對接至衛福部電子病歷伺服器。

2. HAPI FHIR JPA Starter 資料庫儲存原理與架構表

在標準的 HAPI FHIR JPA 伺服器架構中，所有資源並非以扁平表儲存，而是使用高度關聯的核心索引架構：

核心資料表名稱 儲存內容與技術原理 重要欄位與屬性
HFJ_RESOURCE 記錄所有 FHIR 資源的原資料 (Metadata)，是整個 FHIR JPA Server 的核心主表。 RES_ID, RES_TYPE, RES_VERSION
HFJ_RES_VER 儲存資源內容的版本紀錄。FHIR 將原始 JSON/XML 內容以 GZIP 壓縮格式以 BLOB 欄位儲存。 RES_TEXT_VC, RES_VER
HFJ_SPIDX_STRING 字串型索引表。用於快速搜尋 Patient.name 等字串屬性，避開對大 XML/JSON 進行全表掃描。 SP_NAME, SP_VALUE_NORMALIZED
HFJ_SPIDX_QUANTITY 數值型索引表。專專用於對應 Observation.valueQuantity 之數值與 UCUM 單位進行快速範圍查詢。 SP_VALUE, SP_SYSTEM, SP_UNITS
HFJ_SPIDX_TOKEN Token 索引表。用於比對代碼對（如：LOINC Code 或身分證字號識別元）。 SP_SYSTEM, SP_CODE

3. 台灣核心實作指引 (TW Core IG) - 病患 Patient Profile

衛福部 TW Core IG 指引規範原文：
「本 Profile 繼承自 base Patient Resource，並額外規定：Patient.identifier (至少包含一個病歷號/身分證字號/健保卡號，基數 1..*)、Patient.name (病患中文或英文姓名，基數 1..*)、Patient.gender (生理性別，基數 1..1) 必須填寫，此為 Must Support 核心欄位以供跨院索引與身分核對。」

對照與合規要件：在我們的地端資料表 patients 中，mrn (病歷號)、name (姓名) 以及 gender (性別) 均設定為 NOT NULL 強制欄位，完全相容於此規定。

4. 台灣核心實作指引 (TW Core IG) - 生命徵象 Observation Profile

衛福部 TW Core IG 指引規範原文：
「對於生命徵象的 Observation Profile，其 status (狀態) 基數為 1..1、code (LOINC 代碼) 基數為 1..1、subject (關聯病人) 基數為 1..1，且 valueQuantity (量測數值與符合國際 UCUM 單位) 為強烈建議填寫的 Must Support 欄位，用以維持數據的一致性與可跨院交換性。」

對照與合規要件：我們將體溫、心率及呼吸數據以 value_numeric 及 value_unit (符合 UCUM 標準，如生命徵象之體溫填寫 Cel) 形式進行分離儲存與 API 交換，以確保跨系統互補一致性。
