地端部署之實體感測硬體，於後端資料庫登錄之 device_id 分流明細：

  • 心率/呼吸: device-radar-01 (MR60-Radar)
  • 跌倒偵測: device-radar-02 (MR60-Fall-Radar)
  • 體溫/姿態/離床: device-camera-01 (E53-Camera)

地端與雲端系統架構與傳輸規範

1. 數據傳輸生命週期與流向圖

本系統以非接觸式雷達與熱感影像相機作為採集源頭，透過 Python 伺服器對接資料庫，最終將整合數據渲染至護理面板： 

                        sequenceDiagram
                            autonumber
                            participant S as 物理感測硬體 (Radar / FLIR E53)
                            participant B as Python 後端 (server.py)
                            participant DB as PostgreSQL 資料庫
                            participant FE as 護理站監控面板 (index.html)

                            S->>B: 1. 傳送 JSON 數據包 (HTTP POST /api)
                            Note over B: 2. 依指標映射標準代碼與硬體識別碼
- 心率/呼吸 -> device-radar-01
- 跌倒警報 -> device-radar-02
- 體溫/姿態/離床 -> device-camera-01
                            B->>DB: 3. 執行關聯式資料表寫入 (INSERT INTO observations)
                            FE->>B: 4. 定期輪詢最新觀測數據 (GET /api)
                            B->>DB: 5. 跨表查詢最新資料 (SELECT JOIN)
                            DB-->>B: 6. 回傳病患、設備與生理觀測資料
                            B-->>FE: 7. 渲染前端分頁與警報 UI

2. 感測器數據上報 API 規範

地端感測器上報時，向後端發送 HTTP POST 請求。其傳輸規格定義如下：

請求端點 (Endpoint): POST /api
Headers: Content-Type: application/json
// 範例 1: 心率數據上報 (數值型) { "patient_id": "patient-01", "metric_type": "heart_rate", "value": 78.0 } // 範例 2: 姿態辨識上報 (狀態型) { "patient_id": "patient-01", "metric_type": "posture", "value": "lying" // 可接受 "lying" (躺著) 或 "standing" (站立) }

3. 台灣核心實作指引 (TW Core IG) - 病患基本資料規範

衛福部 TW Core IG 原話指出：
「本 Profile 繼承自 base Patient Resource，並額外規定：Patient.identifier (至少包含一個病歷號/身分證字號/健保卡號，基數 1..*)、Patient.name (病患中文或英文姓名，基數 1..*)、Patient.gender (生理性別，基數 1..1) 必須填寫，此為 Must Support 核心欄位以供跨院索引與身分核對。」
🔗 參考官方連結：台灣核心實作指引病患 Profiles 規範

4. 台灣核心實作指引 (TW Core IG) - 生命徵象觀測規範

衛福部 TW Core IG 原話指出：
「對於生命徵象的 Observation Profile，其 status (狀態) 基數為 1..1、code (LOINC 代碼) 基數為 1..1、subject (關聯病人) 基數為 1..1，且 valueQuantity (量測數值與符合國際 UCUM 單位) 為強烈建議填寫的 Must Support 欄位，用以維持數據的一致性與可跨院交換性。」
