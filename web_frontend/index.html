衛生福利部標準規格與 HAPI FHIR 資料庫儲存研析指引

1. HAPI FHIR JPA Starter 資料庫儲存原理與架構表

在標準的 HAPI FHIR JPA 伺服器架構中，所有資源並非以扁平表儲存，而是使用高度關聯的核心索引架構：

核心資料表名稱 儲存內容與技術原理 重要欄位與屬性
HFJ_RESOURCE 記錄所有 FHIR 資源的 Metadata，是整個 FHIR JPA Server 的核心主表。 RES_ID (PK), RES_TYPE (資源類型), RES_VERSION (目前版本)
HFJ_RES_VER 儲存資源內容的版本紀錄。FHIR 將原始 JSON/XML 內容以 GZIP 壓縮格式以 BLOB 欄位儲存。 RES_TEXT_VC (GZIP 壓縮後之 JSON BLOB), RES_VER (版本號)
HFJ_SPIDX_STRING 字串型索引表。用於快速搜尋 Patient.name 等字串屬性，避開對大 XML/JSON 進行全表掃描。 SP_NAME, SP_VALUE_NORMALIZED
HFJ_SPIDX_QUANTITY 數值型索引表。專用於對應 Observation.valueQuantity 之數值與 UCUM 單位進行快速範圍查詢。 SP_VALUE, SP_SYSTEM, SP_UNITS
HFJ_SPIDX_TOKEN Token 索引表。用於比對代碼對（如：LOINC Code 或身分證字號識別元）。 SP_SYSTEM, SP_CODE

2. 台灣核心實作指引 (TW Core IG) - 病患 Patient Profile

衛福部 TW Core IG 指引規範原文：
「本 Profile 繼承自 base Patient Resource，並額外規定：Patient.identifier (至少包含一個病歷號/身分證字號/健保卡號，基數 1..*)、Patient.name (病患中文或英文姓名，基數 1..*)、Patient.gender (生理性別，基數 1..1) 必須填寫，此為 Must Support 核心欄位以供跨院索引與身分核對。」

對照與合規要件：在我們的地端資料表 patients 中，mrn (病歷號)、name (姓名) 以及 gender (性別) 均設定為 NOT NULL 強制欄位，完全相容於此規定。

3. 台灣核心實作指引 (TW Core IG) - 生命徵象 Observation Profile

衛福部 TW Core IG 指引規範原文：
「對於生命徵象的 Observation Profile，其 status (狀態) 基數為 1..1、code (LOINC 代碼) 基數為 1..1、subject (關聯病人) 基數為 1..1，且 valueQuantity (量測數值與符合國際 UCUM 單位) 為強烈建議填寫的 Must Support 欄位，用以維持數據的一致性與可跨院交換性。」

對照與合規要件：我們將體溫、心率及呼吸數據以 value_numeric 及 value_unit (符合 UCUM 標準，如生命徵象之體溫填寫 Cel) 形式進行分離儲存與 API 交換，以確保跨系統互補一致性。
